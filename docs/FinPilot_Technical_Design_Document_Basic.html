1. Introduction

1.1 Purpose

This Technical Design Document describes the design of FinPilot, an Android personal finance management application developed for the SE3092 assignment. It converts the scenario of Kavindu Silva into a technical solution using Kotlin, Jetpack Compose, MVVM, Firebase Authentication, Cloud Firestore, Room, Retrofit, WorkManager, and Koin.

1.2 Application Overview

Kavindu earns from a fluctuating salary, freelance projects, USD-based AdSense revenue, and crypto activity. He struggles to understand total monthly income, committed versus discretionary spending, and whether he is moving toward his LKR 490,000 MacBook Pro M4 goal. FinPilot addresses these problems by combining multi-source income recording, fast expense logging, dashboard analytics, offline-first transaction persistence, and goal tracking.

1.3 Scope

  • Firebase Authentication with email/password, password reset, and Google Sign-In.
  • Multi-currency income entry with live LKR conversion.
  • Quick expense entry with committed and discretionary categorisation.
  • Dashboard totals for income, expenses, net balance, spending overview, and recent transactions.
  • MacBook-oriented savings goal tracking and manual savings boosts.
  • Room-first local transaction persistence with background Firestore sync.

2.2 Core Insight

Kavindu's problem is not only missing records. It is financial invisibility. He lacks a simple place where irregular income, spontaneous expenses, and goal progress become visible together. The system therefore prioritises fast capture, normalised LKR calculations, and an at-a-glance dashboard over complex budget planning.

3. Problem Understanding

3.1 Income Structure

The system must understand that income is not fixed monthly salary only. It must support different sources, different currencies, and irregular arrival dates. Each income entry stores the raw amount, original currency, exchange rate used at entry time, and converted LKR amount. This makes totals historically stable and avoids recalculating past income with a new exchange rate.

3.2 Expense Friction

Kavindu abandoned previous tools because they required too much effort. FinPilot's expense flow uses quick amount chips, category icons, date shortcuts, and optional notes. This reduces the number of decisions needed when logging a transaction in daily life.

3.3 Goal Visibility

The MacBook goal should not be a separate spreadsheet. It is shown in the home/dashboard experience and can receive manual boosts or savings routed from income. This makes goal progress emotionally and financially visible.

3.4 Minimum Self-Awareness Views

  • Net balance summary.
  • Total income and total expenses.
  • Spending categories, including committed and discretionary categories.
  • Recent transactions.
  • Goal progress and remaining amount.

7. MVVM Architecture

+-----------------------------+ | UI Layer - Jetpack Compose | | Login, Register, Home, | | Income, Expense, Goal, | | Profile, History | +--------------+--------------+ | v +-----------------------------+ | ViewModel Layer | | AuthViewModel | | IncomeViewModel | | ExpenseViewModel | | DashboardViewModel | | GoalDetailViewModel | | ProfileViewModel | +--------------+--------------+ | v +-----------------------------+ | Repository Layer | | AuthRepository | | FinanceRepository | | DashboardRepository | | GoalRepository | +--------------+--------------+ | v +-----------------------------+ | Data Sources | | Firebase Auth | | Firestore | | Room Database | | Retrofit Exchange API | +-----------------------------+

The UI layer only displays state and forwards user events. ViewModels hold StateFlow objects and call repositories using viewModelScope. Repositories contain business and data-access logic. The data sources are Firebase Authentication, Firestore, Room, and Retrofit.

Example: Login Flow

  1. User enters email and password in LoginScreen.
  2. LoginScreen calls AuthViewModel.login(email, password).
  3. AuthViewModel sets AuthState.Loading.
  4. AuthRepository calls FirebaseAuth.signInWithEmailAndPassword.
  5. Repository loads the user profile from Firestore.
  6. AuthViewModel emits AuthState.Success or AuthState.Error.
  7. LoginScreen observes the state and navigates to Home or shows an error.

12. Local Room Database and Offline Strategy

Room is used as the local persistence layer. The current Room database contains transaction and recurring expense tables. The key offline design is Room-first writes: when income or expense is saved, the record is inserted locally first and Firestore sync happens asynchronously.

12.1 Room Entities

EntityPurposeImportant Fields
TransactionStores income and expense records locally.id, userId, type, amount, currency, exchangeRate, baseAmountLKR, category, timestamp, isSynced.
RecurringExpenseStores recurring expense templates.id, userId, amount, category, interval, nextExecutionDate, isEnabled.

12.2 Sync Flow

  1. User saves income or expense.
  2. FinanceRepository inserts the transaction into Room.
  3. A background coroutine attempts to upload the transaction to Firestore.
  4. If Firestore succeeds, the local row is marked isSynced = true.
  5. If network fails, the row remains unsynced.
  6. SyncWorker retries unsynced transactions when connectivity is available.
This solves Kavindu's need to log expenses quickly even when mobile signal is weak, such as during travel.

13. Security Rules and Data Ownership

Firestore rules are submitted in firestore.rules. The current rule structure allows authenticated users to access only their own profile document, and only transactions/goals whose userId matches their Firebase Auth UID.

users/{userId} allow read/write if request.auth.uid == userId transactions/{transactionId} allow create if request.auth.uid == request.resource.data.userId allow read/update/delete if request.auth.uid == resource.data.userId goals/{goalId} allow create if request.auth.uid == request.resource.data.userId allow read/update/delete if request.auth.uid == resource.data.userId

13.1 Security Rationale

Client-side checks are not enough because a malicious client can call Firestore directly. Server-side rules provide the final enforcement layer. Since financial records are private, every record must be bound to the authenticated user's UID.

13.2 Future Improvement

Rules can be strengthened by validating required fields, allowed enum values, numeric ranges, and immutable owner fields. This would further reduce accidental or malicious invalid writes.

19. Design Justifications

19.1 MVVM with Repository

MVVM suits Compose because UI recomposes from state. ViewModels expose StateFlow, repositories hide Firebase/Room/API details, and composables remain focused on rendering and events.

19.2 Flat Transactions Collection

Income and expense records are stored in a shared transactions collection. This supports one chronological feed and simple dashboard totals. The type field distinguishes income from expense.

19.3 Store baseAmountLKR

Storing baseAmountLKR avoids recalculating historical values and keeps dashboard aggregation fast. The exchangeRate field preserves the conversion context used at the time of entry.

19.4 Room-first Persistence

Room-first writes make the app responsive and offline-tolerant. Firestore sync happens later without blocking the user's immediate logging action.

19.5 Koin Dependency Injection

Koin provides shared dependencies such as FirebaseAuth, FirebaseFirestore, Room DAOs, repositories, and ViewModels, reducing manual object creation and improving testability.

20. Implementation Status and Limitations

  • Forgot password and Google Sign-In.

  • Room local transactions and recurring expense templates.
  • Income flow with live currency conversion.

20.2 Known Limitations

  • Exchange-rate responses are not cached locally yet.
  • Firestore rules currently check ownership but do not deeply validate field types.
  • Some ViewModel wiring can be cleaned further for full DI consistency.
  • Unit and instrumented test coverage is minimal.

These limitations do not remove the core value of the application, but they are useful to mention honestly during a technical discussion.

21. Conclusion

FinPilot is designed as a scenario-specific personal finance app for Kavindu Silva. It focuses on the problems that caused previous tools to fail: multi-source income, invisible discretionary spending, low tolerance for manual entry, and a savings goal that was not connected to everyday behaviour.

The implementation uses a modern Android stack: Kotlin, Jetpack Compose, Navigation Compose, MVVM, StateFlow, Coroutines, Firebase Authentication, Firestore, Room, Retrofit, WorkManager, and Koin. The architecture separates UI, ViewModels, repositories, and data sources while supporting offline-first transaction capture and background cloud sync.

The system therefore meets the main technical constraints of the SE3092 assignment and provides a credible product that could improve Kavindu's financial clarity within the first week of use.
